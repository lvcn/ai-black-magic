# AI 需求分析与 PRD 生成实战手册

> **适用工具**：Cursor / ChatGPT / Claude / GitHub Copilot / Windsurf / Cline
>
> **定位**：本文件是 AI 助手执行需求分析和 PRD 文档生成任务的系统级指令文件（Skill File）。AI 在处理任何需求相关任务前必须完整阅读本文件，并严格按照其中的流程、模板和规范执行。

>
> **目标读者**：通过 AI 辅助进行需求分析和产品设计的开发者、产品经理或创业者。AI 阅读后应具备资深产品总监级别的需求拆解和文档输出能力。

# 第一部分：需求分析流程

## 1. 需求接收与澄清

### 1.1 需求来源分类

AI 在接收到需求描述后，首先判断需求来源类型，不同来源的处理策略不同：

| 来源类型 | 特征 | 处理策略 |
|---------|------|---------|
| **老板/甲方一句话需求** | "做个类似小红书的东西" | 必须反问澄清，至少追问 5 个问题 |
| **用户反馈** | "这个按钮太难找了" | 追溯到底层诉求，不要直接改 UI |
| **竞品跟进** | "竞品上了这个功能我们也要" | 先分析竞品数据，确认是否值得做 |
| **数据驱动** | "转化率从 3% 跌到 1.5%" | 先定位问题环节，再定解决方案 |
| **技术驱动** | "这个架构需要重构" | 翻译成业务价值，否则排不上优先级 |

### 1.2 需求澄清必问清单

AI 在接收到任何需求时，**必须**确认以下信息。如果用户没有提供，需要主动追问：

**核心五问（缺一不可）：**

1. **目标用户是谁？** — 不接受"所有人"这种回答，必须具体到人群画像
2. **要解决什么问题？** — 区分"真实痛点"和"想象中的需求"
3. **成功的衡量标准是什么？** — 必须有可量化的指标（DAU、转化率、完成率等）
4. **有没有时间节点？** — 影响 MVP 范围和技术选型
5. **有没有预算/资源约束？** — 决定能做到什么程度

**进阶追问（视情况选择）：**

6. 现在用户是怎么解决这个问题的？（现有替代方案）
7. 市面上有没有类似产品？做得怎么样？（竞品情况）
8. 这个需求做了之后，谁最开心？（利益相关方）
9. 如果只能做一个核心功能，你选哪个？（优先级试探）
10. 有没有绝对不能做的事？（红线和约束）

### 1.3 需求澄清输出格式

每次完成需求澄清后，AI 必须输出以下结构化摘要：

```markdown
## 需求澄清摘要

**需求名称**：[一句话标题]
**需求来源**：[老板/用户反馈/竞品/数据/技术]
**提出时间**：[日期]

### 核心信息
- **目标用户**：[具体人群描述]
- **核心问题**：[用户的什么问题没有被解决]
- **期望效果**：[做了之后预期达到什么效果]
- **成功指标**：[可量化的衡量标准]
- **时间要求**：[截止日期或里程碑]
- **资源约束**：[预算/人力/技术限制]

### 待确认项
- [ ] [还需要确认的问题1]
- [ ] [还需要确认的问题2]

### 初步风险
- [识别出的风险1]
- [识别出的风险2]
```

## 2. 用户与场景分析

### 2.1 用户画像构建

AI 必须为每个目标用户群体构建画像，不能笼统地说"用户"：

```markdown
## 用户画像：[画像名称]

**基础信息**
- 年龄段：
- 职业：
- 城市级别：
- 收入水平：
- 技术水平：[小白 / 普通 / 熟练 / 专家]

**行为特征**
- 使用场景：[什么时候、什么地方、为什么会用]
- 使用频率：[每天 / 每周 / 偶尔]
- 使用时长：[单次多长时间]
- 核心诉求：[最想解决的问题]
- 替代方案：[现在怎么解决的]

**心理特征**
- 最在意：[价格 / 效率 / 体验 / 安全]
- 最讨厌：[什么会让他流失]
- 付费意愿：[愿意为什么付费]
```

### 2.2 场景分析框架

使用 **"人 + 时 + 地 + 事 + 感"** 五要素描述每个使用场景：

| 要素 | 描述 | 示例 |
|------|------|------|
| **人** | 谁在使用 | 25 岁女性白领 |
| **时** | 什么时间 | 午休 12:30，刷手机 |
| **地** | 什么场景 | 办公室，网络环境好 |
| **事** | 做什么事 | 想找附近好吃的外卖 |
| **感** | 什么感受 | 选择困难，希望有人推荐 |

### 2.3 用户旅程地图

对于核心流程，AI 必须绘制用户旅程地图：

```markdown
## 用户旅程：[流程名称]

| 阶段 | 用户行为 | 触点 | 情绪 | 痛点 | 机会点 |
|------|---------|------|------|------|--------|
| 认知 | 听朋友推荐 | 朋友圈/口碑 | 好奇 | 不知道是什么 | 分享机制 |
| 了解 | 搜索/下载 | 应用商店 | 期待 | 描述不够直观 | 截图+视频 |
| 首次使用 | 注册/体验 | App | 犹豫 | 注册太麻烦 | 一键登录 |
| 使用 | 完成核心动作 | 核心页面 | 满足/沮丧 | 流程不顺 | 引导优化 |
| 留存 | 再次打开 | 推送/习惯 | 依赖 | 没有新东西 | 个性化推荐 |
| 传播 | 推荐给朋友 | 分享功能 | 自豪 | 没有分享动力 | 邀请奖励 |
```

## 3. 需求分类与优先级排序

### 3.1 需求分类体系

所有需求必须按以下维度分类：

**按类型分：**

| 类型 | 说明 | 示例 |
|------|------|------|
| **功能需求** | 系统应该做什么 | 用户可以用手机号注册 |
| **非功能需求** | 系统应该达到什么标准 | 页面加载时间 < 2 秒 |
| **业务规则** | 业务约束条件 | 每人每天限购 3 件 |
| **数据需求** | 需要什么数据、怎么展示 | 首页展示近 7 天销量 Top 10 |
| **运营需求** | 运营人员需要什么能力 | 后台可以配置 Banner 图 |

**按层次分（卡诺模型）：**

| 层次 | 说明 | 处理方式 |
|------|------|---------|
| **基本型** | 没有就不及格，有了也不加分 | 必须做，不用讨论 |
| **期望型** | 做得越好满意度越高 | 重点投入，差异化竞争 |
| **兴奋型** | 用户没想到，做了超预期 | 选做，用来制造口碑 |
| **反向型** | 做了反而降低满意度 | 坚决不做 |

### 3.2 优先级排序方法

AI 对需求排优先级时，必须使用以下方法之一：

**方法一：四象限法（推荐日常使用）**

```
          高价值
            │
   ┌────────┼────────┐
   │  重要    │  重要    │
   │  不紧急   │  且紧急   │
   │ (P2)    │ (P0)    │
   ├────────┼────────┤
   │  不重要   │  不重要   │
   │  不紧急   │  但紧急   │
   │ (P3)    │ (P1)    │
   └────────┼────────┘
            │
          低价值
   不紧急 ──────────── 紧急
```

**方法二：RICE 评分法（推荐做规划时使用）**

```
RICE 分数 = (Reach × Impact × Confidence) / Effort

Reach（影响面）：这个功能影响多少用户/月
Impact（影响度）：0.25 / 0.5 / 1 / 2 / 3
Confidence（信心度）：100% / 80% / 50%
Effort（工作量）：人天
```

**方法三：MoSCoW 法（推荐做 MVP 时使用）**

| 级别 | 含义 | 占比建议 |
|------|------|---------|
| **Must have** | 没有就不能上线 | 60% |
| **Should have** | 很重要但可以稍后做 | 20% |
| **Could have** | 锦上添花 | 15% |
| **Won't have** | 这次不做 | 5% |

### 3.3 优先级输出格式

```markdown
## 需求优先级排序

| 序号 | 需求名称 | 类型 | 优先级 | RICE 分数 | 预估工时 | 备注 |
|------|---------|------|--------|----------|---------|------|
| 1 | 手机号一键登录 | 功能 | P0 Must | 450 | 3 人天 | 注册转化率瓶颈 |
| 2 | 商品搜索 | 功能 | P0 Must | 380 | 5 人天 | 核心路径 |
| ... | | | | | | |
```

## 4. 竞品分析方法

### 4.1 竞品选择原则

- **直接竞品**：2-3 个，和你做一样的事
- **间接竞品**：1-2 个，不同方式解决同一问题
- **跨界参考**：1 个，其他领域的优秀体验

### 4.2 竞品分析维度

AI 进行竞品分析时，必须覆盖以下维度：

```markdown
## 竞品分析：[竞品名称]

### 基本信息
- 产品名称：
- 所属公司：
- 上线时间：
- 预估用户量：
- 主要盈利模式：

### 功能对比矩阵

| 功能模块 | 我们 | 竞品A | 竞品B | 竞品C |
|---------|------|-------|-------|-------|
| 核心功能1 | 🔨计划 | ✅有 | ✅有 | ❌无 |
| 核心功能2 | 🔨计划 | ✅有 | ❌无 | ✅有 |
| 差异化功能 | 🔨计划 | ❌无 | ❌无 | ❌无 |

### 竞品优劣势

| 维度 | 竞品做得好的 | 竞品做得差的 | 我们的机会 |
|------|-----------|-----------|-----------|
| 产品体验 | | | |
| 功能完整度 | | | |
| 运营策略 | | | |
| 技术实现 | | | |

### 关键结论
1. [我们的差异化方向]
2. [必须对齐的功能]
3. [可以避开的坑]
```

# 第二部分：需求拆解

## 5. 用户故事编写规范

### 5.1 标准格式

所有需求最终必须拆解为用户故事，使用以下标准格式：

```
作为 [角色]，
我想要 [功能/行为]，
以便于 [价值/收益]。
```

**反面示例（不合格）：**
- ❌ "做一个登录功能" — 没有角色、没有价值
- ❌ "作为用户，我想要登录" — 缺少价值说明
- ❌ "作为所有人，我想要所有功能" — 太笼统

**正面示例（合格）：**
- ✅ "作为新用户，我想要用微信一键登录，以便于不用记密码就能快速开始使用"
- ✅ "作为店铺老板，我想要看到今日实时营业额，以便于及时调整促销策略"

### 5.2 用户故事拆分原则

**INVEST 原则：**

| 原则 | 含义 | 检查方法 |
|------|------|---------|
| **I**ndependent | 故事间尽量独立 | 能否独立开发和发布？ |
| **N**egotiable | 可以协商细节 | 有没有过度规定实现方式？ |
| **V**aluable | 对用户有价值 | 用户会不会为此买单？ |
| **E**stimable | 可以估算工作量 | 开发能给出工时吗？ |
| **S**mall | 足够小 | 能在一个迭代内完成吗？ |
| **T**estable | 可以测试验证 | 怎么证明做完了？ |

### 5.3 拆分技巧

**大故事拆小的 5 种方法：**

1. **按用户角色拆**："用户可以管理订单" → "买家可以查看订单" + "买家可以取消订单" + "卖家可以发货"
2. **按操作步骤拆**："用户可以下单" → "选商品" + "填地址" + "选支付方式" + "确认支付"
3. **按业务规则拆**："计算运费" → "满 88 免运费" + "偏远地区加收" + "自提免运费"
4. **按数据变化拆**："商品管理" → "新增商品" + "编辑商品" + "上下架" + "删除商品"
5. **按异常场景拆**："支付" → "正常支付" + "支付超时" + "余额不足" + "重复支付"

## 6. 验收标准定义

### 6.1 验收标准格式

每个用户故事必须附带验收标准，使用 **Given-When-Then** 格式：

```markdown
### 验收标准

**正常流程：**
- Given 用户已登录且商品有库存
- When 用户点击"立即购买"
- Then 跳转到订单确认页，显示商品信息、价格和收货地址

**异常流程：**
- Given 用户已登录但商品无库存
- When 用户点击"立即购买"
- Then 显示"该商品已售罄"提示，按钮变为"到货通知"
```

### 6.2 验收标准检查项

每条验收标准必须通过以下检查：

- [ ] 是否可以直接转化为测试用例？
- [ ] 是否覆盖了正常流程？
- [ ] 是否覆盖了至少 2 个异常流程？
- [ ] 是否描述了边界值？（最大、最小、为空）
- [ ] 是否描述了用户可见的结果？（不是技术实现）

## 7. 边界条件与异常场景

### 7.1 必须考虑的边界场景

AI 在分析任何功能需求时，**必须主动检查**以下边界条件：

**数据边界：**
| 场景 | 要考虑的问题 |
|------|-----------|
| **空数据** | 列表为空时显示什么？搜索无结果时怎么办？ |
| **单条数据** | 只有一条记录时布局是否正常？ |
| **海量数据** | 10 万条数据时性能如何？要不要分页？ |
| **极值** | 金额为 0？数量为 999999？标题 500 字？ |
| **特殊字符** | 输入 emoji、HTML 标签、SQL 注入会怎样？ |

**状态边界：**
| 场景 | 要考虑的问题 |
|------|-----------|
| **首次使用** | 没有任何数据时引导什么？ |
| **未登录** | 哪些功能需要登录？未登录看到什么？ |
| **权限不足** | 没权限时是隐藏按钮还是禁用？ |
| **网络异常** | 断网、弱网时怎么处理？ |
| **并发操作** | 两人同时编辑同一条数据怎么办？ |

**时间边界：**
| 场景 | 要考虑的问题 |
|------|-----------|
| **跨天** | 23:59 下单，00:01 付款，算哪天？ |
| **超时** | 支付超时、验证码过期、Session 过期 |
| **时区** | 有没有海外用户？服务器用什么时区？ |
| **节假日** | 节假日规则是否不同？（如发货时效） |

### 7.2 异常场景输出格式

```markdown
## 异常场景清单：[功能名称]

| 编号 | 异常场景 | 触发条件 | 期望处理方式 | 优先级 |
|------|---------|---------|------------|--------|
| E01 | 网络超时 | 请求超过 10 秒无响应 | 显示重试按钮 | P0 |
| E02 | 库存不足 | 下单时库存为 0 | 提示售罄 + 到货通知 | P0 |
| E03 | 重复提交 | 用户连续点击 2 次 | 防重处理，只提交一次 | P1 |
| ... | | | | |
```

## 8. 业务规则提取

### 8.1 业务规则分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **约束规则** | 必须满足的条件 | 手机号必须 11 位、年龄必须 ≥ 18 |
| **计算规则** | 数值怎么算 | 订单金额 = 商品价 × 数量 - 优惠 + 运费 |
| **状态流转** | 状态怎么变 | 订单：待支付 → 已支付 → 已发货 → 已完成 |
| **触发规则** | 什么情况下触发什么动作 | 库存 < 10 件时发预警通知 |
| **权限规则** | 谁能做什么 | 只有管理员可以删除用户 |

### 8.2 业务规则输出格式

```markdown
## 业务规则清单：[模块名称]

### BR-001：订单金额计算规则
- **规则描述**：订单实付金额 = 商品总价 - 优惠金额 + 运费
- **商品总价** = Σ(单价 × 数量)
- **优惠金额**：按优惠券面额扣减，不能超过商品总价
- **运费**：满 88 元免运费，未满按 8 元计算，偏远地区 +12 元
- **最低金额**：实付金额最低为 0.01 元（不允许为 0）
- **精度**：所有金额精确到分，使用四舍五入

### BR-002：订单状态流转规则
```
待支付 ─→ 已支付 ─→ 待发货 ─→ 已发货 ─→ 已签收 ─→ 已完成
  │          │                      │          │
  ↓          ↓                      ↓          ↓
已取消    退款中 ─→ 已退款       拒收 ─→ 退回中    退货退款
```
- 待支付超过 30 分钟自动取消
- 已支付后 7 天未发货可申请退款
- 签收后 7 天内可申请退货退款
```

## 16. MVP 范围界定

### 16.1 MVP 原则

- **MVP 不是缩水版**，而是用最小成本验证核心假设
- **只留核心路径**：用户完成一次完整的核心价值体验即可
- **砍功能不砍质量**：功能可以少，但做的功能要做好

### 16.2 MVP 范围决策框架

```markdown
## MVP 范围决策
```

# 第六部分：需求质量保障

## 20. 需求评审检查清单

### 20.1 PRD 完整性检查

AI 在输出最终 PRD 前，必须逐项检查：

**整体检查：**
- [ ] 项目背景和目标是否清晰？
- [ ] 成功指标是否可量化？
- [ ] 用户画像是否具体到人群？
- [ ] MVP 范围是否明确？
- [ ] 上线计划是否合理？

**功能检查（每个功能）：**
- [ ] 是否有用户故事？
- [ ] 是否有完整的主流程？
- [ ] 是否有验收标准？
- [ ] 是否覆盖异常场景？
- [ ] 是否考虑了权限差异？
- [ ] 是否考虑了空状态？
- [ ] 是否考虑了数据边界？

**技术检查：**
- [ ] 是否评估了技术可行性？
- [ ] 是否识别了技术风险？
- [ ] 是否有工作量估算？

**合规检查：**
- [ ] 是否需要用户协议/隐私政策？
- [ ] 是否需要实名认证？
- [ ] 是否需要 ICP 备案？
- [ ] 个人信息收集是否合规？

### 20.2 常见需求缺陷检测

AI 在审查需求文档时，必须检测以下常见缺陷：

| 缺陷类型 | 特征 | 修复方法 |
|---------|------|---------|
| **模糊表述** | "系统应该比较快" "支持大量数据" | 改为具体数字：< 2 秒、100 万条 |
| **遗漏异常** | 只写了正常流程 | 补充至少 3 个异常场景 |
| **隐含假设** | "用户肯定会..." | 明确写出假设，加上验证方式 |
| **需求冲突** | A 功能说可以，B 功能说不行 | 找出冲突，确认以哪个为准 |
| **镀金需求** | "顺便加个 AI 推荐" | 质疑必要性，不在 MVP 就砍掉 |
| **技术语言** | "用 Redis 做缓存" | PRD 不写技术方案，只写业务需求 |
